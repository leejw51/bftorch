# PyTorch Sandbox — JSON Protocol (v1)

Backend: Python + PyTorch + FastAPI on `http://127.0.0.1:8200` (configurable via the Makefile `PORT`).
Frontend: TypeScript + Three.js (Vite dev server, proxied to backend).
All communication is JSON over HTTP (REST) and WebSocket. No HTML is generated by the backend.

CORS: backend must allow origin `http://localhost:5173` (Vite) — or just allow `*` for dev.

---

## REST

### `GET /api/health`
→ `200`
```json
{ "status": "ok", "torch": "2.12.0", "device": "cpu", "mps": true }
```

### `GET /api/lessons`
→ `200`
```json
{ "lessons": [
  { "id": "tensors-101", "title": "1. Tensors 101",
    "explanation": "markdown string",
    "starter_code": "python string",
    "hint": "string" }
] }
```

### `POST /api/run`
Request:
```json
{ "code": "x = torch.arange(6).reshape(2,3)\nprint(x)\nx.sum()" }
```
→ `200`
```json
{
  "ok": true,
  "stdout": "tensor([[0,1,2],[3,4,5]])\n",
  "error": null,
  "result_repr": "tensor(15)",
  "duration": 0.012,
  "tensors": [
    { "name": "x", "shape": [2,3], "dtype": "int64",
      "data": [[0,1,2],[3,4,5]], "truncated": false }
  ]
}
```
- `error` is a string (traceback) or `null`.
- `tensors`: every top-level variable in the user namespace that is a torch.Tensor (or numpy array),
  AFTER execution. Convert to nested JSON lists. Cap total elements (e.g. ≤ 4096); if larger,
  send a flattened/truncated sample and set `"truncated": true`. `dtype` is a string. 1-D/2-D/3-D
  are the interesting cases for Three.js; higher dims may be truncated.

## WebSocket — live training

### `WS /ws/train`
Client → server (first message): a JSON config:
```json
{ "demo": "linreg", "epochs": 200, "lr": 0.05, "hidden": 32, "seed": 0 }
```
`demo` ∈ the keys returned by `GET /api/demos`.

### `GET /api/demos`
```json
{ "demos": [
  { "key": "linreg", "label": "Linear Regression", "description": "...",
    "viz": "regression" },
  { "key": "sine", "label": "Fit a Sine Wave (MLP)", "description": "...",
    "viz": "regression" },
  { "key": "classify_blobs", "label": "2-Class Classifier", "description": "...",
    "viz": "classification" }
] }
```

Server → client: a stream of `frame` messages, then it closes the socket.
```json
{
  "type": "frame",
  "epoch": 40, "total": 200,
  "loss": 0.0123,
  "metric": 0.95, "metric_name": "accuracy",   // metric may be null
  "loss_history": [1.2, 0.9, 0.5, ...],         // cumulative, for the 3D loss ribbon
  "log": "epoch 40/200  loss 0.0123  acc 0.950",
  "done": false,
  "viz": { ...see below... }
}
```
Throttled to ~40–60 frames total. Final frame has `"done": true`.
On error, server sends `{ "type": "error", "message": "..." }` then closes.

### `viz` payload by kind

**regression** (linreg, sine):
```json
{ "kind": "regression",
  "x":     [ ... ],    // input scalars, sorted ascending
  "y":     [ ... ],    // ground-truth targets
  "y_pred":[ ... ] }   // current model prediction at each x
```

**classification** (classify_blobs):
```json
{ "kind": "classification",
  "points": [[x0,y0],[x1,y1], ...],   // 2-D inputs
  "labels": [0,1,0, ...],             // class per point
  "grid": {
    "xs":   [ ... ],                  // G values spanning x-range (e.g. G=40)
    "ys":   [ ... ],                  // G values spanning y-range
    "probs":[[...],[...], ...]        // G×G prob of class-1 in [0,1], rows indexed by ys
  } }
```

The frontend Three.js layer decides how to render each kind (3-D loss ribbon from `loss_history`,
data scatter + fitted curve for regression, a height/colored surface from `grid.probs` for
classification). The backend never dictates rendering — it only sends numbers.
